### Функциональные требования
- **F01**: Поиск и бронирование свободных парковочных мест.
- **F02**: Безналичная оплата парковки через систему.
- **F03**: Уведомления о состоянии бронирования и времени парковки.
- **F04**: Интерактивная карта парковок с актуальной информацией о свободных местах.
- **F05**: История транзакций и парковок для пользователя.
- **F06**: Управление тарифами и акциями для администраторов парковки.
- **F07**: Статистика и отчеты для администраторов.

### Нефункциональные требования
#### Производительность (Performance)
- **PER01**: Время отклика интерфейса на действия пользователя (нажатие кнопок, открытие модальных окон) не должен превышать 200 миллисекунд.
- **PER02**: Время ответа на поиск и отображение парковок на карте не должно превышать 2 секунды при 95% запросов.
- **PER03**: Процесс бронирования или оплаты должен завершаться в течение 3 секунд.
- **PER04**: Время выгрузки среднего аналитического отчета для администратора не должно превышать 5 секунд.

#### Масштабируемость (Scalability)
- **SCA01**: Архитектура должна поддерживать горизонтальное масштабирование (возможность добавления новых серверов/инстансов) для обработки растущей нагрузки и пиков активности.
- **SCA02**: На момент запуска система должна стабильно обслуживать до 10 000 активных пользователей
- **SCA03**: Через 3 года система должна быть способна обслуживать до 50 000 активных пользователей

#### Доступность (Availability)
- **AVA01**: Требуется обеспечить доступность системы на уровне не менее 99.9% (uptime).
- **AVA02**: Допускается плановое технологическое окно для обслуживания не чаще одного раза в месяц с 02:00 до 04:00 по московскому времени, с предварительным уведомлением пользователей.

#### Безопасность и аудит (Security)
- **SEC01**: Необходимо обеспечить защиту персональных и платежных данных пользователей в соответствии с современными стандартами (шифрование данных при передаче и хранении).
- **Требования:**
  - TLS 1.3 для передачи данных
  - AES-256-GCM для шифрования данных в покое
  - Хранение данных на территории РФ (требование ФЗ-152)
- **SEC02**: Система должна соответствовать законодательству по обработке персональных данных:
  - **Федеральный закон № 152-ФЗ** "О персональных данных"
  - **Постановление Правительства № 1119** "Об утверждении требований к защите персональных данных при их обработке в информационных системах персональных данных"
  - **Приказ ФСТЭК № 21** "Об утверждении состава и содержания организационных и технических мер по обеспечению безопасности персональных данных"
- **SEC03**: Взаимодействие с платежными шлюзами должно соответствовать стандарту безопасности **PCI DSS (Payment Card Industry Data Security Standard)** версии 4.0.
- **SEC04**: Требуется вести неизменяемый журнал аудита всех критичных действий в соответствии с:
  - **Федеральный закон № 149-ФЗ** "Об информации, информационных технологиях и о защите информации"
  - **ГОСТ Р 57580.1-2017** "Безопасность финансовых операций"
- **SEC05**: Для операций с электронными денежными средствами необходимо соблюдение:
  - **Федеральный закон № 161-ФЗ** "О национальной платежной системе"
  - **Положение Банка России № 683-П** "О порядке осуществления расчетов"

#### Надежность (Durability, Disaster Recovery)
- **DUR01**: Система должна быть отказоустойчивой; сбой одного из компонентов не должен приводить к полной неработоспособности сервиса.
- **DUR02**: Должно производиться ежедневное автоматическое резервное копирование базы данных с хранением бэкапов в географически удаленном хранилище **на территории РФ**.
- **DUR03**: Система должна поддерживать процедуры автоматического восстановления после сбоев для минимизации времени простоя (RTO < 30 минут).

#### Мониторинг (Monitoring)
- **MON01**: Необходимо внедрить систему комплексного мониторинга для отслеживания технических (нагрузка на CPU/RAM, время ответа API, ошибки) и бизнес-метрик (количество бронирований, успешных платежей, DAU/MAU).
- **MON02**: Для реализации мониторинга предлагается использовать стек: Prometheus для сбора метрик, Grafana для визуализации, ELK Stack для централизованного сбора логов и Sentry для отслеживания ошибок.

#### Организационные и юридические вопросы (Legal, Regulatory and Compliance)
- **LRC01**: Система должна соответствовать законодательству в области:
  - **Гражданский кодекс РФ** (глава 37 "Подряд", глава 39 "Возмездное оказание услуг")
  - **Закон РФ № 2300-1** "О защите прав потребителей"
  - **Кодекс РФ об административных правонарушениях** (статьи, касающиеся платных парковок)
  - **Региональные нормативные акты** города Москвы (если применимо):
    - Закон города Москвы № 45 "О платных городских парковках в городе Москве"
    - Постановление Правительства Москвы № 289-ПП "Об организации платных городских парковок"
- **LRC02**: Публичная оферта и пользовательское соглашение должны соответствовать:
  - **Статья 435 ГК РФ** "Оферта"
  - **Статья 438 ГК РФ** "Акцепт"
  - **Федеральный закон № 63-ФЗ** "Об электронной подписи" (для дистанционного заключения договоров)
- **LRC03**: Для рассылки уведомлений соблюдать:
  - **Федеральный закон № 38-ФЗ** "О рекламе" (для маркетинговых сообщений)
  - **Правила рассылки сообщений** операторов связи

#### Локализация и интернационализация (I18n, L10n)
- **L10N01**: Архитектура должна поддерживать многоязычный интерфейс.
- **L10N02**: На первом этапе требуется поддержка русского и английского языков.
- **L10N03**: Форматы дат, времени и валют должны соответствовать региональным настройкам пользователя.

#### Целевые платформа и устройства пользователей
- **PLT01**: Требуется адаптивный веб-интерфейс для корректного отображения на десктопных и мобильных устройствах.
- **PLT02**: Поддержка последних двух мажорных версий браузеров Chrome, Firefox, Safari.
- **PLT03**: Нативные приложения для iOS (версия 15+) и Android (версия 9+).

### Ограничения
- **LIM01**: На этапе запуска сервис работает только в пределах одного города (например, Москва).
- **LIM02**: Интеграция на первом этапе ограничена одним утвержденным платежным шлюзом.
- **LIM03**: Команда разработки для MVP: 6 человек (1 PM, 2 Backend, 2 Frontend/Mobile, 1 QA).
- **LIM04**: Бюджет на разработку MVP составляет 1 млн рублей.
- **LIM05**: Срок разработки MVP и запуска: 6 месяцев.

### Принципы
- **PRN01**: Интерфейс должен быть интуитивно понятным и простым (принцип User-Centered Design).
- **PRN02**: Архитектура должна быть построена на основе микросервисов для обеспечения гибкости, масштабируемости и возможности интеграции с внешними системами.
- **PRN03**: Соблюдение принципа **Privacy by Design** - защита данных закладывается на этапе проектирования системы.
- **PRN04**: Соответствие принципу **Proportionality** - меры безопасности должны быть пропорциональны рискам.
